```
┌──────────────────┐     chrome.runtime.sendMessage      ┌──────────────────┐
│   popup/          │ ──────────────────────────────────→ │  content_scripts/ │
│   index.html/js   │    (START_SCROLL / START_AI_SCROLL) │  index.js         │
```

## 二、完整运行流程（从点击"开始"按钮到结束）

### 2.1 用户点击"开始"按钮

**入口：** `popup/index.js` → `scrollButton` 点击事件 → `startAutoScroll()`

```
用户点击"开始" → startAutoScroll()
                    │
                    ├─ 检查是否已选择岗位
                    ├─ 设置 isRunning = true, matchCount = 0
                    ├─ 保存运行状态到 chrome.storage
                    ├─ 通知 content_script 显示广告
                    │
                    ├─ 判断当前模式
                    │   ├─ AI模式: 发送 START_AI_SCROLL 消息
                    │   │   携带: positionName, jobDescription, aiConfig,
                    │   │         matchLimit, scrollDelayMin/Max, clickFrequency,
                    │   │         enableSound, communicationConfig
                    │   │
                    │   └─ 免费模式: 发送 START_SCROLL 消息
                    │       携带: keywords, excludeKeywords, isAndMode,
                    │             matchLimit, scrollDelayMin/Max, clickFrequency,
                    │             enableSound, communicationConfig
                    │
                    └─ handleConnectionError() 监听连接异常
```

### 2.2 Content Script 接收消息并启动循环

**处理：** `content_scripts/index.js` → `chrome.runtime.onMessage.addListener`

```
收到消息 → switch(message.action)
            │
            ├─ START_SCROLL (免费模式)
            │   ├─ 设置 currentParser.aiMode = false
            │   ├─ 更新 filterSettings（关键词/排除词/与或模式等）
            │   └─ 调用 startAutoScroll()
            │
            ├─ START_AI_SCROLL (AI模式)
            │   ├─ 设置 currentParser.aiMode = true
            │   ├─ 更新 aiSettings（岗位说明/AI配置/沟通配置等）
            │   └─ 调用 startAutoScroll()
            │
            └─ 其他: STOP_SCROLL, UPDATE_KEYWORDS, SETTINGS_UPDATED 等
```

### 2.3 startAutoScroll() — 主循环入口

```javascript
async function startAutoScroll() {
    // 1. 防重入检查
    if (isRunning) return;

    // 2. AI模式配置校验
    if (currentParser?.aiMode) {
        // 必须有岗位说明
        if (!currentParser?.aiSettings?.jobDescription?.trim()) {
            → 报错，return
        }
    }

    // 3. 初始化运行参数
    isRunning = true;
    lastProcessedPosition = 0;

    // 从 parser 获取设置（AI模式用 aiSettings，免费模式用 filterSettings）
    matchLimit = parser.aiMode ? aiSettings.matchLimit : filterSettings.matchLimit;
    scrollDelayMin = parser.aiMode ? aiSettings.scrollDelayMin : filterSettings.scrollDelayMin;
    scrollDelayMax = parser.aiMode ? aiSettings.scrollDelayMax : filterSettings.scrollDelayMax;

    // 4. 滚动到顶部
    window.scrollTo(0, 0);

    // 5. 开始执行滚动循环
    executeScroll();
}
```

### 2.4 executeScroll() — 核心滚动循环

```
executeScroll() ──→ 无限递归循环（直到 isRunning=false 或 matchCount>=matchLimit）
    │
    ├─ 1. 获取所有文档对象（包括iframe）
    │      getAllDocuments() → [document, iframe.contentDocument, ...]
    │
    ├─ 2. 查找候选人卡片元素
    │      使用 currentParser.selectors.items 类名查找
    │      支持精确匹配 + 模糊匹配([class*="xxx"])
    │
    ├─ 3. 过滤出未处理的元素（位置 > lastProcessedPosition）
    │
    ├─ 4. 如果有未处理元素:
    │      ├─ 滚动到该元素位置(smooth)
    │      ├─ 创建临时高亮样式(橙色虚线边框)
    │      ├─ 调用 processElement(element, doc)  ← 核心处理
    │      ├─ 清理高亮
    │      ├─ 等待500ms
    │      └─ 递归调用 executeScroll()  ← 处理下一个
    │
    └─ 5. 如果没有未处理元素:
           ├─ 猎聘网(hliepin): 检查是否需要翻页
           │   └─ shouldNavigateToNextPage() → clickNextPageButton()
           └─ 默认: 向下滚动200px → 递归调用 executeScroll()
```

### 2.5 processElement() — 单个候选人处理流程

这是最核心的函数，流程较为复杂：

```
processElement(element, doc)
    │
    ├─ 1. checkMessageTip() — 检查消息提示（目前Boss已禁用）
    │
    ├─ 2. 检查 matchCount >= matchLimit → 停止
    │
    ├─ 3. extractCandidates([element]) — 从卡片提取候选人基本信息
    │      返回 candidates 数组
    │
    ├─ 4. 遍历每个 candidate:
    │      │
    │      ├─ 4a. getSimpleCandidateInfo(candidate) — 获取简单信息字符串
    │      │       返回：姓名/年龄/学历/学校/经验/期望薪资等文本
    │      │
    │      ├─ 4b. 第一次AI决策：是否查看候选人详情
    │      │   ├─ AI模式: performAIClickDecision(simpleCandidateInfo)
    │      │   │   ├─ 构建prompt（岗位要求 + 候选人基本信息）
    │      │   │   ├─ 调用 sendDirectAIRequest() → https://siliconflow.a.58it.cn/api/chat.php
    │      │   │   ├─ 解析AI返回的JSON: {"decision":"是/否","reason":"原因"}
    │      │   │   └─ 返回 {isok, msg, cost}
    │      │   │
    │      │   └─ 免费模式: currentParser.shouldClickCandidate() — 按概率决定
    │      │
    │      ├─ 4c. 如果决定查看详情(clickCandidate=true):
    │      │   │
    │      │   ├─ clickCandidateDetail(element) — 点击打开详情弹框
    │      │   ├─ randomDelay() — 随机等待
    │      │   │
    │      │   ├─ queryColleagueContactedInfo() — 查询同事是否已沟通过
    │      │   │
    │      │   ├─ extractCandidates2(candidate) — 提取详细信息（第二次）
    │      │   │   （Boss返回null，其他平台可能返回更丰富信息）
    │      │   │
    │      │   ├─ 第二次AI决策：是否打招呼
    │      │   │   ├─ Boss直聘: 如果第一次AI通过，直接打招呼（不再二次决策）
    │      │   │   ├─ 其他平台: performAIClickDecision(详细候选人信息)
    │      │   │   └─ 免费模式: filterCandidate(候选人信息文本) — 关键词匹配
    │      │   │
    │      │   └─ closeDetail() — 关闭详情弹框
    │      │
    │      ├─ 4d. 如果决定打招呼(shouldContact=true):
    │      │   │
    │      │   ├─ 检查 matchCount >= matchLimit → 停止
    │      │   ├─ clickMatchedItem(element) — 点击打招呼按钮
    │      │   ├─ collectPhoneWechatResume() — 可选：索要手机号/微信号/简历
    │      │   │   ├─ 点击"继续沟通"按钮
    │      │   │   ├─ clickPhoneButton() — 点击索要联系方式
    │      │   │   └─ 关闭确认弹框
    │      │   ├─ matchCount++
    │      │   ├─ playNotificationSound() — 播放提示音(notification2.mp3)
    │      │   ├─ processCommunication() — 处理沟通（目前仅猎聘实现了）
    │      │   └─ sendMessage(MATCH_SUCCESS) — 通知popup匹配成功
    │      │
    │      └─ 4e. 如果不打招呼:
    │          └─ addHighlightReason("未打招呼") — 标记为灰色
    │
    └─ 继续处理下一个候选人
```

### 2.6 停止运行

```
用户点击"停止" → stopAutoScroll()
                    ├─ isRunning = false
                    ├─ 清除滚动定时器
                    ├─ 重置 lastProcessedPosition = 0
                    ├─ 清除所有高亮样式
                    ├─ 发送 STOP_SCROLL 到 content_script
                    ├─ 发送 REMOVE_ADS 移除广告
                    └─ 保存状态到 chrome.storage
```

### 4.3 各平台特殊差异

#### Boss直聘 (boss.js) — 最完整的实现

- **数据拦截机制**：通过 `boss_interceptor.js` 在 document_start 注入，Hook fetch 和 XHR API
- 拦截目标URL：`/wapi/zprelation/interaction/bossGetGeek`、`/wapi/zpjob/rec/geek/list` 等
- 拦截到的数据通过 `window.postMessage` 发送给 boss.js，缓存到 `chrome.storage.local.bossZhipinCandidates`
- **候选人数据来自API拦截**而非DOM解析，信息更完整：geekCard、geekWorks、geekEdu等
- getSimpleCandidateInfo 返回极其详细的信息（工作经历、教育经历、自我介绍等）
- **Boss的AI决策策略**：第一次AI通过后直接打招呼，不需要第二次AI决策
- extractCandidates2 返回 null（因为第一次已获取足够信息）
- 支持 resume_downloader 简历下载

#### 猎聘网 h.liepin (hliepin.js) — 支持翻页

- 唯一实现 `shouldNavigateToNextPage()` 和 `clickNextPageButton()` 的平台
- 当页面所有候选人处理完后，自动点击下一页继续

#### 58同城 (employer58.js) — 最简化的处理

- 不需要查看候选人详情（clicked=false 直接 shouldContact=true）
- 所有AI决策都返回 `{isok: true, msg: "58无法筛选"}`
- 免费模式下也直接 shouldContact=true

#### 猎聘网 lpt.liepin (liepin.js) — 有 processCommunication

- 实现了 processCommunication，构建包含公司信息+岗位信息+候选人信息的沟通内容

## 五、AI 筛选流程详解

### 5.1 AI 请求流程

```
performAIClickDecision(simpleCandidateInfo)
    │
    ├─ 特殊处理：58同城直接返回 {isok: true}
    │
    ├─ showAIDecisionModal() — 显示"AI正在思考..."动画
    │
    ├─ 构建请求:
    │   prompt = currentParser.aiSettings.aiConfig.clickPrompt  (系统提示词)
    │   userContent = "岗位要求：{jobDescription}\n候选人基本信息：{candidateInfo}"
    │
    ├─ sendDirectAIRequest(prompt, aiConfig)
    │   ├─ 检查余额 >= 0.1
    │   ├─ 获取手机号(hr_assistant_phone)
    │   ├─ POST https://siliconflow.a.58it.cn/api/chat.php
    │   │   body: { phone, messages: [system, user] }
    │   └─ 返回 { success, response, cost }
    │
    ├─ hideAIDecisionModal()
    │
    └─ parseAIResponse(response, cost)
        ├─ 尝试匹配JSON: {"decision":"是","reason":"符合基本要求"}
        ├─ 决策为"是" → {isok: true, msg, cost}
        └─ 决策为"否" → {isok: false, msg, cost}
```

### 5.2 AI Prompt 模板

```
你是一个资深的HR专家。请根据候选人的基本信息判断是否值得查看其详细信息。

重要提示：
1. 这个API仅用于岗位与候选人的筛选
2. 请根据岗位要求判断是否值得查看这位候选人的详细信息
3. 必须返回JSON格式，包含decision和reason两个字段
4. decision字段只能是"是"或"否"
5. reason字段是决策原因，10个字以内
...

岗位要求：
${岗位信息}

候选人基本信息：
${候选人信息}
```

### 5.3 两次AI决策的时机

| 决策点 | 时机         | 输入           | Boss行为             | 其他平台行为 |
| ------ | ------------ | -------------- | -------------------- | ------------ |
| 第1次  | 不打开详情前 | 简单候选人信息 | AI决策               | AI决策       |
| 第2次  | 打开详情后   | 详细候选人信息 | **跳过，直接打招呼** | AI决策       |

## 六、消息通信机制

### 6.1 Popup → Content Script（通过 chrome.tabs.sendMessage）

| action                | 数据                                                    | 说明           |
| --------------------- | ------------------------------------------------------- | -------------- |
| START_SCROLL          | keywords, excludeKeywords, isAndMode, matchLimit, ...   | 启动免费模式   |
| START_AI_SCROLL       | positionName, jobDescription, aiConfig, matchLimit, ... | 启动AI模式     |
| STOP_SCROLL           | -                                                       | 停止运行       |
| UPDATE_KEYWORDS       | 关键词设置                                              | 实时更新关键词 |
| SETTINGS_UPDATED      | 完整设置                                                | 设置变更通知   |
| SHOW_ADS              | -                                                       | 显示广告       |
| REMOVE_ADS            | -                                                       | 移除广告       |
| COMMUNICATION_PROCESS | companyInfo, jobInfo, communicationConfig               | 沟通处理配置   |

### 6.2 Content Script → Popup/Background（通过 chrome.runtime.sendMessage）

| type          | 数据                                   | 说明                                   |
| ------------- | -------------------------------------- | -------------------------------------- |
| LOG_MESSAGE   | {message, type}                        | 日志消息（info/success/error/warning） |
| MATCH_SUCCESS | {name, age, education, matchTime, ...} | 匹配成功通知                           |

### 6.3 Port连接机制

content_scripts/index.js 通过 `chrome.runtime.connect` 建立长连接（Port），用于持续通信：

- 连接名：`content-script-connection`
- sendMessage 函数封装了最多3次重试逻辑
- 连接断开时自动重连

## 七、数据拦截机制（Boss直聘独有）

### 7.1 注入策略

```json
// manifest.json
{
  "matches": ["*://*.zhipin.com/*"],
  "js": ["content_scripts/sites/boss_injector.js"],
  "run_at": "document_start", // 在页面任何脚本执行前注入
  "all_frames": true
}
```

### 7.2 拦截流程

```
boss_injector.js (document_start)
    └→ 注入 boss_interceptor.js 到页面上下文
        └→ Hook window.fetch 和 XMLHttpRequest
            └→ 拦截目标API响应
                └→ window.postMessage({source:'boss-plugin', type:'geek-list', data})
                    └→ boss.js initDataInterceptor() 接收
                        └→ processInterceptedData() → 缓存到 chrome.storage
```

## 十四、新版本架构讨论

### 14.1 核心改动一：按功能/平台拆分文件

旧项目把大量逻辑堆在 `content_scripts/index.js` 和各平台的单个文件里，新版按功能模块和平台区分文件和文件夹。

新版文件结构：

```
content_scripts/
├── index.js              ← 调度器（单循环，统一编排主动+被动流程）
├── injector.js           ← 注入脚本（3个方法：通信、查找、点击）
├── strategies/           ← 流程策略（方式A：每个平台一个函数）
│   ├── boss.js           ← Boss的主动流程 + 被动流程
│   ├── lagou.js          ← 拉勾的主动流程 + 被动流程
│   ├── liepin.js         ← 猎聘的主动流程 + 被动流程
│   ├── hliepin.js        ← 猎聘(h)的主动流程 + 被动流程
│   ├── zhilian.js        ← 智联的主动流程 + 被动流程
│   ├── employer58.js     ← 58同城的主动流程 + 被动流程
│   └── grasswave.js      ← 草浪的主动流程 + 被动流程
└── config/
    └── selectors.js      ← 各平台的选择器配置
```

### 14.2 核心改动二：注入脚本只保留3个方法

注入到招聘网站的脚本极致精简，只提供3个原子能力，所有业务逻辑都在插件侧完成：

| 方法                     | 职责       | 说明                                                         |
| ------------------------ | ---------- | ------------------------------------------------------------ |
| `sendMessage(data)`      | 与插件通信 | `chrome.runtime.sendMessage` / `onMessage` 等消息收发        |
| `findElements(selector)` | 查找元素   | `querySelector` / `querySelectorAll` 等 DOM 查找，带重试逻辑 |
| `clickElement(selector)` | 点击元素   | `element.click()` 等 DOM 操作，带等待                        |

设计原则：

- 注入脚本是纯粹的"手和眼"，只做执行，不做决策
- 平台的 selector 配置外置到 `config/selectors.js`，通过消息传给注入脚本
- 各平台的流程差异通过 `strategies/*.js` 中的策略函数来编排（方式A：写成代码）
- Boss直聘的 API 拦截器（`boss_interceptor.js` + `boss_injector.js`）暂不纳入此方案，后续单独处理

示例——插件侧通过消息驱动注入脚本：

```
插件侧 → 注入脚本: { action: "find", selector: ".candidate-card-wrap" }
注入脚本 → 插件侧: { result: [元素信息...] }

插件侧 → 注入脚本: { action: "click", selector: ".btn-greet", index: 0 }
注入脚本 → 插件侧: { result: true/false }
```

### 14.3 核心改动三：新增被动回复消息功能

#### 为什么不合入主循环

- 主动打招呼是**滚一个处理一个**的串行流程，如果插入消息检测会打断节奏
- 用户可能想**只开被动回复，不开主动打招呼**——合在一起做不到
- 主循环的停止/启动逻辑会变复杂（主动打招呼停了，被动回复还要继续吗？）
- 页面只有一个，两个循环同时操作DOM一定会冲突（抢页面）

#### 采用方案：统一调度器 + 消息优先

虽然主动和被动不能真正并行执行（页面只有一个），但检测和决策可以交错进行，通过统一调度器保证同一时刻只有一个操作在执行：

```
┌─────────────────────────────────────────────┐
│              统一调度器 (Orchestrator)         │
│                                             │
│   ┌─────────┐        ┌─────────────────┐    │
│   │ 主动任务  │        │  被动任务         │    │
│   │ 生产者   │        │  生产者           │    │
│   └────┬────┘        └────┬────────────┘    │
│        │                  │                 │
│        ↓                  ↓                 │
│   ┌──────────────────────────────────┐      │
│   │        任务队列 (Queue)           │      │
│   │  [find, find, click, find, ...]  │      │
│   └──────────────┬───────────────────┘      │
│                  │                           │
│                  ↓  逐个执行，同一时刻只做一件事 │
│   ┌──────────────────────────────────┐      │
│   │        执行器 → 注入脚本(3个方法)   │      │
│   └──────────────────────────────────┘      │
└─────────────────────────────────────────────┘
```

#### 优先级设计

被动回复消息比主动打招呼优先级更高（候选人在等回复，不及时回复体验差）：

| 任务类型         | 优先级 | 说明                     |
| ---------------- | ------ | ------------------------ |
| 被动：检测新消息 | 最高   | 每轮都要检查，快速       |
| 被动：回复消息   | 高     | 检测到消息后优先处理     |
| 主动：打招呼流程 | 普通   | 没有消息需要回复时才继续 |

#### 主循环逻辑

```javascript
async function mainLoop() {
  while (isRunning) {
    // 1. 优先检测消息
    if (hasNewMessage()) {
      await handleNewMessage(); // 被动回复流程
      continue;
    }

    // 2. 没有消息时，处理主动打招呼的下一步
    await processNextStep();
  }
}
```

每一轮循环先检测消息，没有消息才继续主动打招呼的下一步，消息优先插队。

#### 被动回复流程

```
被动回复流程 (handleNewMessage)
    │
    ├─ 1. findElements(消息提示选择器)  — 检测是否有未读消息
    │      例如 Boss 的小红点、猎聘的未读标记
    │
    ├─ 没有未读 → 返回，继续主动流程
    │
    └─ 有未读:
        ├─ 2. clickElement(消息入口)    — 点击进入对话
        │
        ├─ 3. findElements(消息列表)     — 读取历史消息
        │      提取: 对方最后一条消息内容 + 时间
        │
        ├─ 4. 通知插件侧 → AI生成回复
        │      输入: 岗位信息 + 对方消息内容 + 沟通配置
        │      输出: 回复文本
        │
        ├─ 5. 点击输入框 → 输入回复文本 → 点击发送
        │      ⚠️ 注意: 这里可能需要扩展注入脚本的方法（如 inputText）
        │
        └─ 6. 返回候选人列表页 → 等待N秒 → 回到主循环
```

#### 待细化问题

1. **输入文本方法**：被动回复需要往输入框填入AI生成的文本，当前3个方法中没有"输入"能力，后续需讨论是扩展为4个方法还是在 `clickElement` 的 options 中支持传 text
2. **Boss API拦截器**：暂不纳入当前方案，后续单独处理
3. **各平台消息入口的 selector 差异**：需要在 `config/selectors.js` 中新增被动回复相关的选择器配置
